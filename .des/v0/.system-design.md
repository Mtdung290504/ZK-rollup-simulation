# Thiết kế Hệ thống: ZK-Rollup Simulator (PoC)

Tài liệu này quy định cấu trúc kỹ thuật, luồng dữ liệu và thiết kế trạng thái cho hệ thống mô phỏng ZK-Rollup, sử dụng cơ chế **Treasury Symmetric Bridge** để tương thích với mạch ZK tĩnh.

---

## 1. Tổng quan Kiến trúc

Hệ thống gồm **4 thành phần độc lập**, giao tiếp qua HTTP/JSON:

### 1.1. Layer 1 Server — Port 3000

Mô phỏng Ethereum Smart Contract. Chịu trách nhiệm:

- Khóa và giải phóng ETH thật (Vault)
- Xác thực ZK Proof và Merkle Proof
- Lưu trữ State Root và DA Root (chỉ lưu hash, không lưu dữ liệu)
- Ghi nhận Deposit Event và Nullifier đã sử dụng

### 1.2. Layer 2 Server — Port 5000

Mô phỏng Sequencer Node. Chịu trách nhiệm:

- Đồng bộ Deposit Event từ L1
- Xử lý giao dịch với Soft Finality
- Quản lý Mempool và Transaction List
- Đóng Batch và gọi SnarkJS sinh ZK Proof
- Publish DA Blob lên Archive Node

### 1.3. Frontend App — Port 8080

Giao diện người dùng. Chịu trách nhiệm:

- Quản lý cặp khóa EdDSA phía client
- Ký giao dịch chuyển tiền L2
- Giao diện Bridge: Nạp tiền và Rút tiền

### 1.4. Archive Node (DA Observer)

Công cụ off-chain lưu trữ dữ liệu giao dịch. Chịu trách nhiệm:

- Nhận và lưu DA Blob từ Sequencer
- Cung cấp dữ liệu để người dùng tự dựng Merkle Proof khi rút tiền

## 2. Thiết kế Cơ sở Dữ liệu

Dữ liệu lưu dưới dạng JSON để dễ debug và quan sát trạng thái.

### 2.1. Layer 1 State — `l1_db.json`

```json
{
	"vault": {
		"0xAliceL1": 50.0,
		"0xBobL1": 100.0
	},
	"bridge_contract": {
		"total_locked_eth": 0.0,
		"current_state_root": "0xINITIAL_STATE_ROOT",
		"batch_history": {
			"1": {
				"state_root": "0xSTATE_ROOT_1",
				"da_root": "0xDA_ROOT_1"
			}
		},
		"pending_deposits": [],
		"claimed_nullifiers": {}
	}
}
```

**Ghi chú các trường:**

- `total_locked_eth` — tổng ETH đang bị khóa, tăng khi Deposit, giảm khi Withdraw.
- `current_state_root` — root mới nhất đã được xác minh bởi ZK Proof.
- `batch_history` — lưu cả `state_root` lẫn `da_root` của từng batch để phục vụ Withdraw verification.
- `pending_deposits` — danh sách Deposit Event chưa được L2 xử lý.
- `claimed_nullifiers` — tập hợp các `nullifier_hash` đã dùng, chống double-spend khi rút tiền.

### 2.2. Layer 2 State — `l2_db.json`

```json
{
	"accounts": {
		"EDDSA_PUB_TREASURY": {
			"balance": 340282366920938463463374607431768211455,
			"nonce": 0,
			"index": 0
		}
	},
	"transactions": [],
	"system": {
		"last_proven_tx_index": -1,
		"last_processed_deposit_id": -1,
		"merkle_tree": {
			"nodes": {}
		}
	}
}
```

**Ghi chú các trường:**

- `EDDSA_PUB_TREASURY` — ví kho bạc, số dư khởi tạo là `2¹²⁸ − 1` (MAX_UINT128). Đóng vai trò hố đen thanh khoản: mọi Deposit đều là Treasury → User, mọi Withdraw đều là User → Treasury. Tổng số dư hệ thống luôn bằng `2¹²⁸ − 1`, không mint thêm.
- `last_proven_tx_index` — con trỏ giao dịch cuối cùng đã được đưa vào batch và prove thành công. Batch tiếp theo lấy các giao dịch từ `last_proven_tx_index + 1`.
- `last_processed_deposit_id` — con trỏ Deposit Event cuối cùng đã được L2 xử lý. Dùng để filter khi sync L1, tránh xử lý lại deposit cũ.

**Quan hệ kế toán Treasury:**

```
total_locked_eth (L1) = MAX_UINT128 - treasury_balance (L2)
```

Khi người dùng nạp 10 ETH: L1 tăng `total_locked_eth` lên 10, L2 giảm `treasury_balance` xuống 10 (Treasury chuyển 10 cho User). Số tổng hai vế luôn bằng `MAX_UINT128`.

## 3. Luồng Dữ liệu

### 3.1. Luồng Nạp tiền (Deposit: L1 → L2)

**Mục đích:** Chuyển ETH từ L1 vào tài khoản L2 của người dùng, thông qua một giao dịch L2 tự động từ Treasury.

**Tại sao cần Treasury:** Mạch ZK chỉ biết xử lý giao dịch EdDSA ký bởi người dùng hợp lệ. Không có cơ chế "mint" trong mạch. Treasury là tài khoản đặc biệt được Sequencer kiểm soát, đóng vai người gửi hợp lệ để đưa tiền vào L2 mà không vi phạm ràng buộc mạch.

**Luồng:**

```
1. User gọi POST /contract/deposit
   → L1 khóa ETH vào Vault
   → L1 ghi Deposit Event vào pending_deposits[] với deposit_id tăng dần

2. L2 gọi GET /contract/deposits/pending
   → Lọc: deposit_id > last_processed_deposit_id
   → Với mỗi deposit mới:
       a. Tạo giao dịch L2: Treasury → User (amount)
       b. Validate giao dịch (format, amount > 0)
       c. Cập nhật state ngay (Soft Finality):
          Treasury balance -= amount
          User balance    += amount
          Treasury nonce  += 1
       d. Append vào transactions[]
       e. Cập nhật last_processed_deposit_id

3. Khi đóng batch:
   → L2 sinh ZK Proof
   → Song song:
       [L2 → L1] POST /contract/batch/submit (proof, state_root, da_root)
       [L2 → Archive] POST /archive/blobs (batch_id, transactions[])
   → L1 verify proof → cập nhật current_state_root và da_root trong batch_history
```

### 3.2. Luồng Chuyển tiền (Transfer: L2 → L2)

**Mục đích:** Chuyển tiền giữa hai tài khoản L2 với xác nhận tức thì (Soft Finality).

**Luồng:**

```
1. User ký giao dịch EdDSA phía client
   → Gọi POST /l2/transfer (to, amount, nonce, signature)

2. L2 validate:
   - Chữ ký EdDSA hợp lệ
   - Nonce khớp với nonce hiện tại của sender
   - sender_balance >= amount + fee

3. Cập nhật state ngay (Soft Finality):
   sender_balance   -= amount + fee
   receiver_balance += amount
   treasury_balance += fee
   sender_nonce     += 1

4. Append vào transactions[]

5. Khi đóng batch: (giống bước 3 của Deposit Flow)
```

**Lưu ý quan trọng:** Vì state cập nhật trước khi có Proof (Soft Finality), Sequencer phải validate lại toàn bộ thứ tự giao dịch trong batch trước khi sinh Proof. Một giao dịch hợp lệ tại thời điểm nhận vào có thể trở nên không hợp lệ sau khi các giao dịch trước nó thay đổi state. Nếu batch fail, Sequencer cần cơ chế rollback hoặc loại bỏ giao dịch vi phạm.

### 3.3. Luồng Rút tiền (Withdraw: L2 → L1)

**Mục đích:** Chuyển ETH từ L2 về L1, thông qua cơ chế gửi vào Treasury trên L2 rồi claim trên L1 bằng Merkle Proof.

**Tại sao gửi vào Treasury:** Treasury là địa chỉ đặc biệt duy nhất mà L1 chấp nhận làm điều kiện để giải phóng ETH. L1 không thể nhìn thấy nội dung L2, nhưng có thể verify Merkle Proof chứng minh rằng một giao dịch "User → Treasury" đã tồn tại trong một batch đã được prove.

**Giai đoạn 1 — Đưa giao dịch lên L2:**

```
1. User ký giao dịch: sender=User, receiver=Treasury, amount=X
   → Gọi POST /l2/transfer
   → Xử lý như Transfer thông thường (validate → update state → append)

2. Giao dịch này được đóng vào một batch
   → Batch được prove và submit lên L1
   → DA Blob được publish lên Archive
```

**Giai đoạn 2 — Claim ETH trên L1:**

```
1. User gọi GET /archive/blobs/:batch_id
   → Archive trả về danh sách transactions[] của batch đó

2. User tự dựng Merkle Proof từ danh sách đó
   (chứng minh giao dịch User → Treasury tồn tại trong DA tree của batch)

3. User gọi POST /contract/withdraw:
   {
     batch_id,
     tx_data: { sender, receiver=Treasury, amount },
     merkle_proof: [...sibling_hashes],
     nullifier_hash: hash(tx_data)
   }

4. L1 xử lý theo thứ tự:
   a. Check nullifier_hash chưa tồn tại trong claimed_nullifiers
      → Nếu đã tồn tại: reject (chống double-spend)
   b. Verify Merkle Proof dựa trên da_root của batch_id trong batch_history
      → Nếu sai: reject
   c. Kiểm tra receiver trong tx_data === Treasury address
      → Nếu không phải: reject
   d. Kiểm tra vault[user_l1_addr] đủ để trả
      → Nếu không đủ: reject (lỗi hệ thống, không nên xảy ra)
   e. Thực hiện:
      - Ghi nullifier_hash vào claimed_nullifiers
      - Giảm total_locked_eth
      - Chuyển ETH từ Vault cho user_l1_addr
```

## 4. API Reference

Quy ước đặt tên:

- `/contract/...` — gọi đến L1 Smart Contract (mô phỏng on-chain call)
- `/l2/...` — gọi đến L2 Sequencer (off-chain)
- `/archive/...` — gọi đến Archive Node (Data Availability)

### 4.1. L1 Server (Port 3000)

| Method | Endpoint                     | Mô tả                                                 |
| ------ | ---------------------------- | ----------------------------------------------------- |
| `POST` | `/contract/deposit`          | Khóa ETH vào Vault, ghi Deposit Event                 |
| `GET`  | `/contract/deposits/pending` | Trả về danh sách Deposit Event chưa xử lý             |
| `POST` | `/contract/batch/submit`     | Nhận Proof + Roots, verify và cập nhật State Root     |
| `POST` | `/contract/withdraw`         | Verify Merkle Proof, giải phóng ETH cho người dùng    |
| `GET`  | `/contract/state`            | Trả về state hiện tại (root, locked_eth, batch_count) |

### 4.2. L2 Server (Port 5000)

| Method | Endpoint            | Mô tả                                                |
| ------ | ------------------- | ---------------------------------------------------- |
| `POST` | `/l2/transfer`      | Nhận giao dịch ký bởi User, validate và update state |
| `POST` | `/l2/batch/force`   | Admin trigger: đóng batch hiện tại và sinh Proof     |
| `GET`  | `/l2/sync-deposits` | Sequencer trigger: đồng bộ Deposit Event từ L1       |
| `GET`  | `/l2/state`         | Trả về state L2 hiện tại (accounts, last indexes)    |

### 4.3. Archive Node

| Method | Endpoint                   | Mô tả                                      |
| ------ | -------------------------- | ------------------------------------------ |
| `POST` | `/archive/blobs`           | Sequencer gửi DA Blob sau khi đóng batch   |
| `GET`  | `/archive/blobs/:batch_id` | User/Tool tải transactions[] của một batch |

## 5. Bất biến Bảo mật

Các thuộc tính sau phải được đảm bảo xuyên suốt hoạt động hệ thống:

**Không mint:** Mạch ZK không có cơ chế tạo tiền. Mọi tiền vào L2 đều từ Treasury, mọi tiền rời L2 đều về Treasury. Treasury balance giảm đúng bằng lượng ETH bị khóa trên L1.

**Data Availability:** Mọi giao dịch trong batch đều được publish lên Archive Node dưới dạng DA Blob trước hoặc song song với việc submit proof lên L1. Người dùng có thể tự verify mà không cần tin tưởng Sequencer.

**Chống double-spend:** L1 lưu `claimed_nullifiers` và kiểm tra trước khi xử lý bất kỳ yêu cầu Withdraw nào. Nullifier được tạo từ hash của nội dung giao dịch, đảm bảo mỗi giao dịch chỉ được claim đúng một lần.

**Tính nhân quả batch:** Giao dịch thứ `i+1` luôn được xử lý trên state do giao dịch thứ `i` để lại. Sequencer phải validate lại toàn bộ thứ tự trước khi sinh Proof để đảm bảo Soft Finality không mâu thuẫn với Hard Finality.

**Liveness qua Padding:** Batch luôn có thể được đóng ngay cả khi số giao dịch thực ít hơn `N_TXS`, nhờ cơ chế padding với `enabled=0` trong mạch ZK.
